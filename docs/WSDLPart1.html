2 Component Model

2.1 Description

2.1.1 The Description Component

WSDL Components and type system components

WSDL Components = interfaces, bindings and services

type system components = element declarations and type definitions

interface operations, endpoints

type definitions are required, interfaces, bindings, services and element declarations are optional

Components can be defined in the WSDL document itself, in included documents or in imported namespaces

To use components from another namespace, wsdl:import must be used.

All QName references must resolve to components

2.1.2 XML Representation of Description Component

wsdl:description element

targetNamespace attribute

children: documentation, import, include, types, interface, binding, service

Two components of different kinds can have the same QName

WSDL documents can include elements and attributes from other namespaces at some designated points

2.1.2.1 `targetNamespace` attribute information item

targetNamespace is in the empty namespace

type xs:anyURI which must be an IRI

2.1.3 Mapping Description's XML Representation to Component Properties

interfaces: document's + imported + included

bindings: document's + imported + included

services: document's + imported + included

element declarations

type definitions: 19 primitive types + 25 derived types + any types defined in the document, imported or included

2.2 Interface

2.2.1 The Interface Component

operation = sequence of input and output messages

interface = set of operations

an interface can extend other interfaces

interface fault components

interfaces are referred to by QName

properties: name, extended interfaces interface faults, interface operations

2.2.2 XML Representation of Interface Component

wsdl:interface

attributes: name, extends, styleDefault

child elements: documentation, fault, operation

2.2.2.1 `name` attribute information item with interface [owner element]

QName of the interface = targetNamespace of the description + name of the interface

The name attribute is in no namespace

Note: attribute do not take the default namespace when a default namespace is declared.

2.2.2.2 `extends` attribute information item

Whitespace-separated list of xs:QName, no namespace.

List of the interfaces that this interface derives from.

Must not contain duplicates

2.2.2.3 `styleDefault` attribute information item

no namespace

Type xs:anyURI

If present, must contain absolute IRIs

Indicate the default style used to construct element declaration of interface message references

2.2.3 Mapping Interface's XML Representation to Component Properties

Name, extended interfaces, interface faults, interface operations

2.3 Interface Fault

2.3.1 The Interface Fault Component

Describes some possible application-specific faults, but not all faults.

Does not describe system faults.

Name, message content model (#any #none, #other, #element), element declaration, parent.

Faults in the same interface cannot have the same name, but faults in different interfaces can have the same name.

Since the interfaces can be in the same namespaces, faults cannot be identified by name or qualified name.

If two faults with the same name are defined on different interfaces within a derivatin chain, they must have the same properties.

Not WSDL documents are usually serialized in XML, but can be serialized in other type systems.

2.3.2: XML Representation of Interface Fault Component

XML element fault within an interface element

Namespace: wsdl

name attribute

element attribute

Other attributes within other namespaces

documentation element child

Other elements within other namespaces

2.3.2.1 `name` attribute information item with `fault` owner element

Attribute name name, type xs:NCName, no namespace

2.3.2.2 `element` attribute information item with `fault` owner element

Attribute name element, no namespace, type xs:QName or xs:token with values #any, #none, #other.

2.3.3 Mapping Interface Fault's XML Representation to Component Properties

Name, message content model, element declaration, parent

Message content element is #element when the element attribute is the QName of an element

2.4 Interface Operation

2.4.1 The Interface Operation Component

WSDL Part 2 defines message exchange patterns.

Name, message exchange pattern, interface message references, interface fault references, style, parent

Interface operation names are not globally unique, and duplicates must be equivalent.

2.4.1.1. Message Exchange Pattern

Placeholder message

Fault propagation ruleset: fault-replaces-message, message-triggers-fault, no-faults.

Message exchange patterns are idenfied by IRIs

2.4.1.2 Operation Style

IRIs that identify operation styles.

See WSDL2 Part 2

2.4.2 XML Representation of Interface Operation Component

Element name operation, namespace wsdl, parent interface

Required name attribute

Optional pattern attribute

Optional style attribute

Optional documentation child elements

Optional input child elements

Optional output child elements

Optional infault child elements

Optional outfault child elements

Other elements from other namespaces

2.4.2.1 `name` attribute information item with `operation` owner element

Attribute name name, no namespace, type xs:NCName

2.4.2.2 `pattern` attribute information item with `operation` owner element

Attribute name pattern, no namespace, type xs:anyURI, must be an absolute IRI.

2.4.2.3 `style` attribute informatin item with `operation` owner element

Attribute name style, no namespace, type xs:anyURI, must be an absolute IRI.

2.4.3 Mapping Interface Operation's XML Representation to Component Properties

name, message exchange pattern, interface message references, interface fault references, style, parent

Default value for message exchange pattern: http://www.w3.org/ns/wsd/in-out

2.5 Interface Message Reference

2.5.1 The Interface Message Reference Component

message label, direction, message content model, element declaration, parent

The message label identifies the role that the message plays in the message exchange pattern.

The direction must be in or out and must match the direction defined in the message exchange pattern.

The message content model is #any, #none, #other or #element.

2.5.2 XML Representation of Interface Message Reference Component

Location: description/interface/operation

Element name: input or output

Namespace: wsdl

Attributes: messageLabel, element, others

Child elements: documentation, others

2.5.2.1 `messageLabel` attribute information item with `input` or `output` owner element

Attribute name messageLabel, type xs:NCName, no namespace

2.5.2.2 `element` attribute information item with `input` or `output` owner element

Attribute name element, no namespace, type xs:QName or xs:token with values #any, #none, #other

2.5.3 Mapping Interface Message Reference's XML Representation to Component Properties

Message exchange pattern: that of the interface.

Direction: in for input and out for output.

The message label is not required, if absent, it is inferred from the message pattern.

message label, direction, message content model, element declaration, parent

2.6 Interface Fault Reference

2.6.1 The Interface Fault Reference Component

fault-replaces-message: the fault replaces the message

messages-triggers-fault: the fault will travel in the opposite direction of the message

interface fault, message label, direction, parent

2.6.2 XML Representation of Interface Fault Reference

Parent element: description/interface/operation

Element names: infault, outfault

Namespace: wsdl

Attributes: ref, messageLabel, others

Child elements: documentation, others

2.6.2.1 `ref` attribute item with `infault`, or `outfault` owner element

Attribute name: ref, no namespace, type xs:QName

2.6.2.2 `messageLabel` attribute information item with `infault`, or `outfault` owner element

Attribute name: messageLabel, no namespace, type: xs:NCName

If the message exchange pattern has more than one fault in the specified direction, the messageLabel attribute must be present.

2.6.3 Mapping Interface Fault Reference's XML Representation to Component Properties

interface fault, message label, direction, parent

2.7 Binding

A binding component describes a concrete message format and transmission protocol

A given binding can be used for multiple interfaces

Bindings can be specified on a per-operation basis.

When the binding specifies operation-specific or fault-specific details, it must specify the interface containing the corresponding operations and faults.

Otherwise, bindings do not need to specify interfaces

WSDL 2 Part 2 defines bindings for SOAP 1.2 and HTTP.

name, interface, type, binding faults, binding operations
